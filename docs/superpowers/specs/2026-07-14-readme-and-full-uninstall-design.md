# README 精简与完整卸载设计

## 目标

精简 README，使首次部署路径更容易扫描；新增一个统一的完整卸载入口，一次清理 Server 和 Agent 的本机安装产物，同时保留源码仓库。

## README 结构

README 保留以下内容：

1. 项目用途和核心能力。
2. 运行要求与代码获取方式。
3. Server 快速安装、必填配置、启动和健康检查。
4. Agent 快速安装、必填配置、启动和上线确认。
5. 常用日志命令、升级命令和完整卸载命令。

重复的依赖安装、状态检查和逐项脚本行为说明合并或删除。配置表只保留会影响首次运行或常见部署方式的选项，完整配置仍可从 `deploy/env/*.env.example` 查看。

## 卸载入口

新增 `deploy/uninstall.sh`，不拆分 Server 和 Agent 模式。脚本每次都清理两端，避免同一台机器曾经安装过不同角色时留下残余。

卸载流程：

1. 除非传入 `--yes` 或 `-y`，先显示将删除配置和数据的警告，并要求明确输入 `y` 或 `yes`。
2. 停止并禁用 `proxypulse-api.service`、`proxypulse-bot.service` 和 `proxypulse-agent.service`。服务未安装或已经停止时继续执行。
3. 删除三个 systemd unit，重新加载 systemd，并清理失败状态。
4. 删除 `/etc/proxypulse` 和 `/var/lib/proxypulse`。
5. 删除当前源码目录内由安装脚本创建的 `.venv` 和默认 SQLite 数据库 `proxypulse.db`。
6. 保留源码目录，并在完成信息中用 shell 安全转义后的绝对路径打印 `sudo rm -rf -- <源码目录>`，由用户自行决定是否执行。

脚本支持 `--help`，未知参数返回非零状态。所有清理操作可重复执行；缺失的服务、目录或文件不应导致卸载失败。

## 数据边界与安全性

“完整卸载”指 ProxyPulse 安装脚本在本机创建的服务、配置、状态、虚拟环境和默认 SQLite 数据。自定义 SQLite 路径、PostgreSQL 数据、反向代理配置、防火墙规则、系统 Python 和源码仓库不由脚本删除。

删除命令固定使用明确路径和 `--` 参数分隔符。源码删除命令只打印、不执行，防止脚本删除自身所在目录。

## 测试与验证

自动化测试在临时目录中复制卸载脚本，并用假的 `sudo` / `systemctl` 命令记录系统级操作，避免修改测试主机。测试覆盖：

- 默认不确认时不删除任何内容。
- `--yes` 会请求清理全部三个服务、两个系统目录，并删除临时仓库中的 `.venv` 与默认数据库。
- 缺失安装产物时仍可成功重复运行。
- 完成输出包含源码保留说明及安全转义的删除命令。
- `--help` 和未知参数行为。

最后运行脚本测试、项目全量测试和 Bash 语法检查，并确认 README 中的路径、参数和命令与实际文件一致。
