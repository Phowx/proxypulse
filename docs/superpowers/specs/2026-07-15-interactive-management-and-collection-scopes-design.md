# ProxyPulse 交互式管理与可选采集设计

日期：2026-07-15  
状态：已确认

## 背景

当前 Server、Agent 的安装脚本和完整卸载脚本彼此独立，安装后仍需手工编辑环境文件。卸载只能同时删除 Server 与 Agent，无法安全地只移除一个角色。Agent 还会固定采集所有主机指标，使用者无法根据隐私、流量统计或最小化采集等需求选择采集范围。

本次改造提供一个统一的交互式管理入口，同时保留现有脚本作为可复用的执行层；并使 Agent 的采集范围可在安装时选择、安装后重新配置。

## 目标

1. 提供一个有说明、可重复执行的一键交互式管理脚本。
2. 将 Server 安装、Agent 安装、采集配置、状态查看和三种卸载方式明确分开。
3. 安装向导直接收集并写入必要配置，减少安装后的手工步骤。
4. 提供标准、隐私、流量和自定义采集方案。
5. 未启用的指标以“未采集”表示，不使用 `0` 冒充真实数据。
6. 保留现有数据，并兼容已有安装和现有自动化脚本。
7. 完全卸载 Server 与 Agent 后打印源码删除命令，但不自动删除源码。

## 非目标

- 不在本次改造中加入 Web 管理后台。
- 不自动配置反向代理、TLS 或 Telegram Bot。
- 不自动删除外部 PostgreSQL 数据库或 Cloudflare 资源。
- 不改变 Agent 接入令牌由 Server/Bot 创建的现有流程。

## 总体架构

新增 `deploy/manage.sh` 作为唯一推荐给用户的交互式入口。现有脚本仍保留，并作为可单独调用的执行层：

- `deploy/install-server.sh`：安装或更新 Server。
- `deploy/install-agent.sh`：安装或更新 Agent。
- `deploy/uninstall.sh`：根据参数卸载 Server、Agent 或全部角色。
- `deploy/lib/*.sh`：共享环境检测、输入校验、配置写入、服务检测和输出逻辑。

交互脚本负责解释、选择和确认；底层脚本负责可重复、可测试的实际变更。这样既能提供一键体验，也不会破坏无交互自动化场景。

## 主菜单

`bash deploy/manage.sh` 显示以下菜单：

1. 安装或更新 Server
2. 安装或更新 Agent
3. 重新配置 Agent 采集范围
4. 查看服务状态
5. 卸载 Server
6. 卸载 Agent
7. 完全卸载 Server 与 Agent
0. 退出

每个选项执行前说明会创建、修改、保留或删除的内容。危险操作必须单独确认，直接按回车不能确认删除。

## 安装向导

### Server

基础向导询问：

- Telegram Bot Token；敏感输入不回显。
- 管理员 Telegram ID；支持逗号分隔的多个纯数字 ID。
- 对外 Server URL；必须是有效的 `http://` 或 `https://` URL。

其余配置沿用当前值或示例默认值。数据库、外部通知、Cloudflare 和报告时间等保留在环境文件中，归为高级配置，不强迫普通安装逐项回答。

向导先收集并验证全部输入，再安装依赖、写入配置、安装 systemd 单元并启动服务。已有配置会作为默认值显示；敏感值只显示“已配置”，不会回显原文。重复运行等同于升级或重新配置。

### Agent

基础向导询问：

- Server URL。
- Agent 名称。
- Server/Bot 生成的接入令牌；敏感输入不回显。
- 采集预设或自定义采集范围。
- 使用网络采集时的接口策略；标准预设默认为自动识别主接口。

配置验证通过后才写入 `agent.env`，然后安装或更新 Agent 并启动服务。

### 依赖

脚本启动时检查 `python3`、`venv`、`pip` 和 `systemctl`。Debian/Ubuntu 缺少 Python 依赖时，解释将安装的软件包，并在确认后执行：

```text
python3 python3-venv python3-pip
```

其他发行版不猜测包管理命令，而是列出缺失项并退出。已有依赖不会重复安装。

## 可选采集范围

### 六个采集组

1. `identity`：主机名、操作系统/平台和非回环本地 IP。
2. `cpu`：CPU 使用率、逻辑核心数和 1 分钟负载。
3. `memory`：内存使用率、已用量和总量。
4. `disk`：根文件系统使用率、已用量和总量。
5. `network`：累计接收/发送流量和所使用的网络接口。
6. `uptime`：运行时长及用于识别计数器重置/主机重启的信息。

### 预设

- 标准：启用 1–6；网络接口策略为 `auto`。
- 隐私：关闭 `identity`，启用其余五组。
- 流量：只启用 `network` 与 `uptime`。
- 自定义：逐项启用或关闭。

启用 `network` 时必须同时启用 `uptime`。向导会自动补充并解释原因。自定义模式至少需要选择一个采集组。

### 配置格式

Agent 环境文件新增一个规范化、逗号分隔的配置：

```text
PROXYPULSE_COLLECTIONS=identity,cpu,memory,disk,network,uptime
```

缺少该配置的旧安装按“标准”处理，确保升级后行为不变。解析时拒绝未知组名，去重并按固定顺序保存。

网络采集沿用现有配置：

- `auto`：自动识别主接口，作为标准默认值。
- `fixed`：使用指定接口。
- `aggregate`：汇总符合条件的接口。

只有启用 `network` 时才询问和校验网络接口策略。

## Agent 上报行为

Agent 的在线心跳独立于指标采集，并按轮询周期持续发送。因此即使只启用 `identity`，或某个采集器临时失败，Server 仍能正确判断 Agent 是否在线。

- 关闭 `identity`：注册/心跳不发送主机名、平台和 IP 信息。
- 关闭某指标组：对应字段发送为 `null` 或不产生指标快照，不发送伪造的零值。
- `cpu`、`memory`、`disk`、`network`、`uptime` 中至少启用一组时，Agent 才发送包含已启用字段的指标快照；只启用 `identity` 时不写入全空快照。
- 单个采集器失败时记录错误并保留心跳；其他已启用采集组继续工作。
- Agent 在注册和心跳中报告当前启用的采集组，Server 保存该范围。

Server/Bot 使用采集范围区分：

- “未启用”：该采集组不在 Agent 的范围中。
- “暂未上报”：采集组已启用，但当前还没有有效样本。

## Server 数据模型与兼容迁移

节点记录新增当前采集范围，用于展示和诊断。注册与心跳请求允许携带采集组列表；旧 Agent 不携带时按标准范围处理。

指标请求中的 CPU、内存、磁盘、负载、流量和运行时长字段改为可空。历史指标表相应列允许 `NULL`：

- SQLite 启动迁移检测旧表约束，只在需要时重建 `metric_snapshots` 表并复制现有数据和索引。
- PostgreSQL 使用 `ALTER COLUMN ... DROP NOT NULL` 放宽约束。
- 迁移不删除或改写已有指标值。

统计逻辑只处理实际采集的数据：

- SQL 聚合自然忽略 `NULL`。
- 流量增量、配额和日报仅在网络累计值与运行时长齐全时计算。
- 缺少网络采集时不产生错误流量或配额消耗。
- Bot 格式化层根据采集范围显示“未启用”或“暂未上报”。

## 配置写入与安全

- 全部输入验证通过后，先写同目录临时文件，再原子替换目标环境文件。
- 已有配置修改前创建带时间戳的备份。
- `/etc/proxypulse/*.env` 权限固定为 `0600`。
- Token、Secret 等敏感值不写入终端输出或命令行参数。
- URL、Telegram ID、采集组和网络接口策略在写入前校验。
- `sudo` 权限在实际修改系统前一次性检查，避免向导末尾才失败。

## 分角色卸载

### 卸载 Server

删除：

- `proxypulse-api.service`
- `proxypulse-bot.service`
- `/etc/proxypulse/server.env`

默认 SQLite 数据库属于 Server 数据，删除前进行第二次独立确认。外部数据库只删除连接配置，不操作数据库本身。

保留 Agent 的服务、配置和状态。Agent 仍存在时保留共享 `.venv` 和源码目录。

### 卸载 Agent

删除：

- `proxypulse-agent.service`
- `/etc/proxypulse/agent.env`
- `/var/lib/proxypulse/agent-state.json`；空状态目录一并清理。

保留 Server 服务、配置和数据库。Server 仍存在时保留共享 `.venv` 和源码目录。

### 完全卸载

删除：

- Server 与 Agent 的全部 systemd 服务。
- `/etc/proxypulse`。
- `/var/lib/proxypulse`。
- 源码目录中的共享 `.venv`。
- 默认 SQLite 数据库。

源码仓库始终保留。全部角色均已移除后，脚本打印经过 shell 转义的源码删除命令，例如：

```text
sudo rm -rf -- /path/to/proxypulse
```

脚本不执行该命令。如果仍有任一角色依赖源码，则不建议删除源码，而是明确说明原因。

分角色卸载也支持从 `deploy/uninstall.sh` 通过显式参数调用；无参数时保持安全的交互提示，不能静默猜测卸载范围。

## 服务状态

状态菜单分别显示 API、Bot 和 Agent：

- 是否已安装 systemd 单元。
- 是否设置为开机启动。
- 当前是否运行。

异常服务同时显示查看日志的命令，不直接输出可能包含敏感配置的完整日志。

## 失败处理

- 依赖安装和 Python 包更新成功后才替换 systemd 单元。
- 配置校验失败时不改动现有文件和服务。
- 服务重启失败时保留配置备份，显示失败服务及对应 `journalctl` 命令。
- 可重复执行安装、卸载和状态检查；不存在的服务或文件不会导致卸载失败。
- 用户中断向导时不写入半成品配置。

## 测试策略

### Shell 脚本

使用临时目录和伪造的 `sudo`、`systemctl`、包管理命令测试：

- 菜单路由及退出。
- Server/Agent 配置生成、默认值、权限和备份。
- 输入校验与用户取消。
- Server、Agent、完全卸载各自的删除和保留边界。
- 共享 `.venv` 的保留与最终清理。
- 完成卸载后的源码删除命令。
- 缺失依赖检测和 Debian/Ubuntu 安装分支。

### Python

覆盖：

- 采集组解析、默认范围、未知值和 `network => uptime` 约束。
- 各采集组合只调用对应采集器。
- 心跳与指标采集解耦。
- 注册、心跳和指标 Schema 的向后兼容。
- SQLite 兼容迁移保留历史数据并允许空指标。
- 节点展示区分“未启用”和“暂未上报”。
- 空网络指标不参与流量、配额和日报计算。

最终执行完整测试套件，并对 Shell 脚本运行语法检查；如果环境提供 ShellCheck，再执行 ShellCheck。
